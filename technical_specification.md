# Формальное техническое задание
## Проект: локальное веб-приложение для преобразования обучающих видео в методические материалы с базой знаний и AI-чатом

**Версия:** 1.0  
**Дата:** 2026-04-22  
**Статус:** Утверждённый черновик для MVP

---

## 1. Общие сведения

### 1.1. Наименование системы
Локальное веб-приложение для обработки обучающих видео, генерации методических материалов, накопления базы знаний и AI-чата по базе знаний.

### 1.2. Основание для разработки
Необходимо разработать инструмент, который позволяет пользователю загружать обучающие видео и автоматически превращать их в структурированные методические материалы с последующим накоплением единой базы знаний и доступом к ней через AI-чат.

### 1.3. Назначение системы
Система предназначена для:
- преобразования обучающих видео в текстовые материалы;
- накопления единой базы знаний из нескольких видео;
- поиска и использования накопленных знаний через AI-чат;
- подготовки материалов в формате, пригодном для последующей обработки, экспорта и обучения.

### 1.4. Формат эксплуатации
- Режим использования: локальный
- Количество пользователей: 1
- Тип интерфейса: веб-интерфейс
- Способ развёртывания: Docker
- Целевая среда: локальная машина пользователя

---

## 2. Цели и ожидаемый результат

### 2.1. Цель проекта
Создать локальное веб-приложение, позволяющее загружать одно обучающее видео, извлекать из него речь и визуальную информацию, формировать структурированные учебные материалы и сохранять их в единую базу знаний с последующим доступом через AI-чат.

### 2.2. Ожидаемый результат
По итогам обработки каждого видео пользователь получает:
- сырую транскрипцию;
- очищенную транскрипцию;
- конспект;
- методичку;
- чек-лист;
- возможность ручной правки каждого этапа;
- возможность добавить материал в общую базу знаний;
- возможность экспортировать отдельный материал и общую базу знаний в DOCX.

Дополнительно пользователь получает:
- AI-чат по всей базе знаний;
- возможность строить базовые визуальные представления информации;
- контроль этапов обработки через интерфейс прогресса.

1. Загрузка одного видеофайла.
2. Поддержка форматов: mp4, mov, avi, mkv, webm.
3. Обработка видео длительностью до 2 часов.
4. Извлечение аудио из видео.
5. Извлечение ключевых кадров из видео.
6. OCR по ключевым кадрам.
7. Транскрибация аудио с использованием OpenAI.
8. Просмотр и ручная правка сырой транскрипции.
9. Автоматическая очистка и нормализация транскрипции.
10. Просмотр и ручная правка очищенной транскрипции.
11. Генерация конспекта.
12. Генерация методички.
13. Генерация чек-листа.
14. Просмотр и ручная правка итоговых материалов.
15. Формирование diff перед добавлением в общую базу знаний.
16. Подтверждение добавления материала в общую базу знаний.
17. Хранение базы знаний в структурированном виде.
18. Экспорт отдельного материала в DOCX.
19. Экспорт общего master-документа в DOCX.
20. AI-чат по всей базе знаний на основе RAG.
21. Режимы чата:
   - строго по документу;
   - с пояснениями ИИ.
22. Вывод фрагментов источников и указание раздела/блока в ответах чата.
23. Страница визуализации.
24. Экспорт визуализации в PNG.
25. Удаление данных пользователем.
26. Отображение прогресса обработки по этапам.

2. Поддержка пакетной загрузки нескольких видео.
3. Многопользовательский режим.
4. Авторизация и разграничение прав.
5. История версий документов.
6. Внешние интеграции (Notion, Google Docs, Telegram и др.).
7. Облачное развёртывание.
8. Продвинутая автоматическая дедупликация знаний.
9. Расширенные шаблоны под разные виды учебных материалов.

## 4. Термины и определения

- **Сырая транскрипция** — текст, полученный напрямую из аудиораспознавания без глубокой стилистической и структурной обработки.
- **Очищенная транскрипция** — нормализованный текст, очищенный от слов-паразитов, воды и хаотичной устной формы.
- **Конспект** — краткое структурированное изложение содержания видео.
- **Методичка** — основной учебный документ, содержащий структурированные разделы, инструкции, примеры, FAQ и выводы.
- **Чек-лист** — краткий перечень действий и правил для практического применения.
- **База знаний** — совокупность всех добавленных материалов, сформированных из обработанных видео.
- **Master-документ** — единый агрегированный документ, содержащий добавленные материалы по всем видео.
- **Chunk** — логический текстовый фрагмент базы знаний, используемый для индексации и поиска.
- **RAG** — подход, при котором ответы AI формируются с использованием релевантных фрагментов базы знаний.

## 5. Общие требования к системе

### 5.1. Требования к архитектуре
Система должна состоять из следующих функциональных блоков:
- frontend-интерфейс;
- backend API;
- модуль обработки видео;
- модуль OCR;
- модуль транскрибации;
- модуль генерации материалов;
- модуль управления базой знаний;
- модуль RAG-поиска и AI-чата;
- модуль визуализации;
- модуль экспорта в DOCX и PNG.

### 5.2. Требования к технологическому стеку
- Frontend: React
- Backend: FastAPI
- Язык backend: Python 3.11
- База данных: SQLite
- AI/LLM/STT/Embeddings: OpenAI
- Контейнеризация: Docker

### 5.3. Требования к хранению данных
Система должна хранить:
- метаданные обработанных видео;
- сырые транскрипции;
- очищенные транскрипции;
- конспекты;
- методички;
- чек-листы;
- OCR-данные;
- структурированные блоки базы знаний;
- чанки для RAG;
- embeddings;
- визуализации.

Система не должна хранить исходные видео после завершения обработки, если пользователь не указал иное.

### 5.4. Внутренний формат данных
Внутренним источником истины должна являться структурированная модель данных, а не DOCX-файл.  
DOCX должен использоваться как формат экспорта и финальной сборки.

## 6. Функциональные требования

## 6.1. Загрузка и валидация видео

### 6.1.1. Общие требования
Система должна:
- принимать один видеофайл за одну операцию обработки;
- поддерживать форматы mp4, mov, avi, mkv, webm;
- валидировать расширение и тип файла;
- контролировать максимально допустимую длительность видео (до 2 часов);
- уведомлять пользователя о некорректных файлах.

### 6.1.2. Результат
После успешной загрузки создаётся запись о задаче обработки и запускается пайплайн обработки.

---

## 6.2. Обработка видео

### 6.2.1. Система должна выполнять
1. Извлечение аудио из видео.
2. Извлечение ключевых кадров.
3. OCR по ключевым кадрам.
4. Подготовку текстовых данных для последующей генерации материалов.

### 6.2.2. Требования к ключевым кадрам
- ключевые кадры должны извлекаться в объёме, достаточном для распознавания основных слайдов и экранов;
- система должна минимизировать избыточные почти одинаковые кадры;
- результаты OCR должны быть связаны с исходной задачей обработки.

---

## 6.3. Транскрибация

### 6.3.1. Общие требования
Система должна:
- транскрибировать аудио с помощью OpenAI;
- формировать сырую транскрипцию;
- сохранять её в системе;
- показывать её пользователю в интерфейсе;
- позволять редактировать вручную.

### 6.3.2. Требования к качеству
- транскрипция должна сохранять смысл исходной речи;
- в случае сомнительных участков пользователь должен иметь возможность вручную скорректировать текст.

---

## 6.4. Очистка и нормализация текста

### 6.4.1. Система должна
- удалять слова-паразиты;
- убирать воду;
- структурировать хаотичную речь;
- повышать читаемость текста;
- сохранять технически значимые фрагменты;
- учитывать текст, распознанный по кадрам.

### 6.4.2. Выходной результат
Очищенная транскрипция должна быть доступна для просмотра, ручной корректировки и дальнейшей генерации материалов.

---

## 6.5. Генерация материалов

### 6.5.1. Конспект
Система должна формировать конспект, содержащий:
- краткое содержание;
- основные идеи;
- ключевые тезисы;
- основные определения и важные замечания.

### 6.5.2. Методичка
Система должна формировать методичку, содержащую:
- заголовок;
- краткое содержание;
- основные разделы;
- пошаговые инструкции;
- тезисы;
- примеры;
- чек-листы;
- FAQ;
- выводы.

### 6.5.3. Чек-лист
Система должна формировать чек-лист, содержащий:
- последовательность действий;
- краткие правила;
- ключевые пункты для практической работы.

### 6.5.4. Требования к стилю
Все материалы должны формироваться в нейтральном учебном стиле, без разговорных вставок, без лишней воды и без выдумывания информации.

### 6.5.5. Ручная правка
Пользователь должен иметь возможность редактировать:
- сырую транскрипцию;
- очищенную транскрипцию;
- конспект;
- методичку;
- чек-лист.

---

## 6.6. Формирование базы знаний

### 6.6.1. Общая логика
Система должна позволять пользователю добавлять обработанные материалы в общую базу знаний.

### 6.6.2. Шаблон блока
Каждый добавляемый блок должен иметь следующую структуру:
- Название видео
- Конспект
- Методичка
- Чек-лист

### 6.6.3. Порядок добавления
- в MVP новые блоки добавляются в конец общей базы знаний;
- автоматическое интеллектуальное распределение по разделам в MVP не требуется.

### 6.6.4. Контроль пользователя
Перед добавлением система должна:
- показать diff того, что будет добавлено;
- позволить вручную отредактировать содержимое;
- выполнить добавление только после явного подтверждения пользователя.

---

## 6.7. Экспорт

### 6.7.1. Система должна поддерживать экспорт
- отдельного материала в DOCX;
- общего master-документа в DOCX.

### 6.7.2. Требования к формированию DOCX
Экспортируемый документ должен:
- иметь читаемую структуру;
- сохранять заголовки и подзаголовки;
- быть пригодным для чтения и дальнейшей ручной работы.

---

## 6.8. AI-чат по базе знаний

### 6.8.1. Общие требования
Система должна предоставлять чат-интерфейс, работающий по всей базе знаний.

### 6.8.2. Режимы работы
Должны поддерживаться два режима:
1. **Строго по документу** — ответ формируется только на основании найденных релевантных фрагментов.
2. **С пояснениями ИИ** — ответ формируется на основании найденных релевантных фрагментов с возможностью дополнительного пояснения.

### 6.8.3. Требования к ответу
Ответ должен содержать:
- основной текст ответа;
- фрагменты использованных источников;
- указание раздела/блока, откуда получена информация.

### 6.8.4. Поддерживаемые сценарии
Чат должен корректно обрабатывать запросы вида:
- найти раздел;
- сравнить темы;
- сформировать чек-лист;
- составить план обучения/тренировок;
- объяснить тему на основе базы знаний.

### 6.8.5. Ограничение галлюцинаций
В режиме “строго по документу” система не должна выдумывать информацию.  
Если релевантных данных недостаточно, система должна сообщить об отсутствии подтверждённой информации в базе знаний.

---

## 6.9. Индексация и поиск

### 6.9.1. Общие требования
Система должна:
- разбивать базу знаний на логические чанки;
- вычислять embeddings для чанков через OpenAI;
- хранить embeddings локально;
- выполнять поиск релевантных чанков по пользовательскому запросу.

### 6.9.2. Требования к чанкам
Каждый chunk должен быть связан с:
- названием видео;
- типом блока (конспект, методичка, чек-лист);
- разделом/подразделом.

---

## 6.10. Визуализация

### 6.10.1. Поддерживаемые типы визуализаций
Система должна поддерживать построение:
- карточек;
- инфографики;
- карты знаний.

### 6.10.2. Экспорт
Визуализация должна экспортироваться в PNG.

### 6.10.3. Статус в MVP
Функциональность визуализации допускается как упрощённая в рамках MVP, но архитектурно должна быть предусмотрена.

---

## 6.11. Удаление данных

Система должна позволять пользователю удалять:
- результаты обработки отдельного видео;
- сформированные материалы;
- связанные embeddings;
- соответствующие блоки базы знаний.

4. транскрибация;

## 9. Нефункциональные требования

### 9.1. Производительность
- система должна быть пригодна для локального запуска;
- пользователь должен видеть прогресс выполнения длительных операций;
- промежуточные результаты должны сохраняться.

### 9.2. Надёжность
- при ошибке на любом этапе система должна возвращать понятное сообщение;
- система не должна терять уже полученные текстовые артефакты при сбое следующего этапа.

### 9.3. Масштабируемость
Для MVP требуется поддержка одного пользователя.  
Масштабирование на многопользовательскую эксплуатацию не является обязательным, но архитектура backend должна не препятствовать развитию проекта.

### 9.4. Безопасность
Специальные требования к защите данных в MVP не предъявляются, кроме:
- локального хранения данных;
- возможности удаления данных пользователем.

---

## 10. Требования к развёртыванию

Система должна запускаться в локальной среде с использованием Docker.  
Конфигурация должна задаваться через переменные окружения.

Минимально должны быть настроены:
- frontend;
- backend;
- локальная база данных;
- тома/каталоги для промежуточных данных и экспорта.

---

## 11. Ограничения и допущения

1. В первой версии используется только загрузка локального видеофайла.
2. Основной язык материалов — русский.
3. OCR выполняется по ключевым кадрам.
4. Внутренний формат хранения должен быть структурированным и пригодным для чанкинга.
5. DOCX является форматом экспорта, а не единственным рабочим представлением данных.
6. Для поиска по базе знаний требуется локальное хранение embeddings и реализация векторного поиска.

---

## 12. Критерии приёмки MVP

MVP считается реализованным, если пользователь может:
1. загрузить видео в поддерживаемом формате;
2. увидеть прогресс обработки по этапам;
3. получить сырую транскрипцию;
4. отредактировать сырую транскрипцию;
5. получить очищенную транскрипцию;
6. отредактировать очищенную транскрипцию;
7. получить конспект, методичку и чек-лист;
8. отредактировать итоговые материалы;
9. увидеть diff перед добавлением в базу знаний;
10. подтвердить добавление в базу знаний;
11. добавить блок в общую базу знаний;
12. экспортировать отдельный материал в DOCX;
13. экспортировать общий master-документ в DOCX;
14. задать вопрос по базе знаний через чат;
15. получить ответ с фрагментами источников и указанием раздела/блока;
16. удалить данные по запросу.
